1. نظرة عامة على المشروع وأهدافه الهندسية

يمثل هذا المستند وثيقة تصميم البرمجيات (Software Design Document - SDD) التفصيلية للأنظمة البرمجية المشتركة: منصة الجامعة الموحدة (UIMP) والنظام الفرعي لإدارة المجموعات البحثية (RGMS). تم بناء النظامين وفق معمارية برمجية متطورة تعتمد على استقلالية الخدمات وفصل الأكواد لضمان أفضل مستويات الحماية والأداء، مع المحافظة على قنوات تواصل مشتركة لضمان تجربة مستخدم موحدة.

1.1 مشكلة الدراسة والحل المقترح

تعاني الأنظمة الجامعية التقليدية من التداخل الشديد بين البيانات الإدارية (مثل سجلات الطلاب ورواتب الموظفين) والبيانات البحثية (مثل براءات الاختراع والمنشورات العلمية). يؤدي هذا التداخل إلى مشاكل أمنية معقدة وصعوبة في صيانة الأنظمة وتحديثها. لذلك، يقترح هذا النظام **معمارية مفصولة هيكلياً ومترابطة أمنياً** بحيث يعمل كل من نظام الجامعة ونظام البحوث كتطبيقات منفصلة (Decoupled Micro-apps) تتشارك قاعدة بيانات علائقية واحدة ونظام تحقق موحد عبر المتصفح.

1.2 نطاق المنظومة الأولى: منصة الجامعة الموحدة (UIMP Core)

يختص هذا النظام بكافة العمليات والبيانات الأكاديمية والإدارية الأساسية داخل الجامعة:

  • سجل الطلاب الأكاديمي: إدارة الملفات الشخصية، رقم القيد الفرعي، التخصص، والبرامج الدراسية، مع ميزة تحديث حالة القيد (ACTIVE, SUSPENDED, GRADUATED).
  • سجل الموظفين وأعضاء هيئة التدريس: إدارة المسميات الوظيفية، البيانات المالية، التوزيع على الكليات والأقسام، والدرجات الأكاديمية.
  • الهيكل التنظيمي للجامعة: إدارة الكليات (Faculties)، الأقسام (Departments)، والبرامج الدراسية (Programs).
  • المنشآت والمرافق: إدارة الفروع الجامعية (Campuses)، المباني (Buildings)، والقاعات الدراسية والمعامل (Rooms).
  • سجلات التدقيق والمراقبة الأمنية: نظام آلي غير قابل للتعديل يسجل كافة العمليات الحساسة في المنظومة لضمان الشفافية.

1.3 نطاق المنظومة الثانية: إدارة المجموعات البحثية (RGMS Subsystem)

يختص هذا النظام بإدارة وتنظيم الأنشطة البحثية والعلمية في الجامعة، ويتكامل بشكل وثيق مع الكوادر الأكاديمية والطلابية الموثقة في النظام الأساسي:

  • المجموعات البحثية (Research Groups): تأسيس المجموعات العلمية، تحديد رئيس المجموعة (Principal Investigator - PI)، وإدارة انتساب الأعضاء ونسبة مساهمتهم.
  • المشاريع البحثية (Research Projects): متابعة المشاريع العلمية، الميزانيات المخصصة لها، جهات التمويل (Funding Sources)، وخطط التقدم والمراحل المنجزة (Milestones).
  • المنشورات وبراءات الاختراع (Publications & Patents): توثيق الأبحاث العلمية المنشورة وبراءات الاختراع وتصنيف المساهمين والمخترعين وترتيبهم.
  • المعدات وحجوزات المعامل (Equipment & Lab Bookings): تسجيل الأجهزة العلمية الثمينة، جدولة صيانتها، وإدارة حجوزات الباحثين لها لتجنب التعارض.

2. البنية المعمارية وهيكلية التواصل (System Architecture)

تعتمد المنظومة على نمط تصميم **التطبيقات المفصولة ذات الولاية المشتركة (Decoupled Monolith with Shared State)**. يتيح هذا النمط تشغيل كل نظام كخادم ويب مستقل تماماً بينما يتشاركان الجلسات وقاعدة البيانات.

2.1 رسم توضيحي لتدفق الجلسات وتسجيل الدخول الموحد (SSO Flow)

يوضح الرسم التالي كيفية عبور المستخدم والتحقق الأمني المشترك بين المنفذين دون تكرار تسجيل الدخول:

[User Browser Session]
       │
       ├─► (Port 8000) ──► [UIMP Core App] ──► Authenticates User ──► Generates Cookie (laravel_session)
       │                                                                      │
       │                                                                Saves Session in
       │                                                                      │
       │                                                                      ▼
       │                                                          [Shared sessions Table]
       │                                                                      ▲
       │                                                                Reads Session
       │                                                                      │
       └─► (Port 8001) ──► [RGMS Subsystem] ◄─────────────────────────────────┘

2.2 آلية عمل التزامن والمشاركة الأمنية (SSO Mechanics)

لتحقيق التزامن الأمني الكامل بين المشروعين، تم تعديل ملفات التكوين والشيفرات البرمجية لتشمل:

  • تطابق مفتاح تشفير الجلسة: تم توحيد قيمة الـ APP_KEY في ملفات الـ .env لكلا المشروعين. هذا التوحيد يضمن لبرمجيات الحماية القدرة على تشفير الجلسة وفك تشفيرها وقراءة توكنات المتصفح بشكل متبادل.
  • سيرفر الجلسات المشترك (Database Session Store): تم إلغاء تخزين الجلسات في الملفات المحلية للمشروع واستخدام خيار قاعدة البيانات SESSION_DRIVER=database. يتم تخزين الجلسات النشطة في جدول sessions في قاعدة البيانات، مما يسمح لنظام البحوث بقراءة صلاحية الجلسة التي تم فتحها في نظام الجامعة الأساسي فوراً.

3. قاموس قاعدة البيانات العلائقية الكامل (Exhaustive Data Dictionary)

تحتوي قاعدة البيانات المشتركة uimp_rgms على الجداول التفصيلية التالية التي تنظم عمل النظامين بالكامل:

3.1 جدول حسابات المستخدمين (users)

يخزن بيانات الدخول والتأمين الأساسية للمستخدمين.

اسم الحقل (Column) نوع البيانات (Type) القيود (Constraints) شرح الحقل
id UUID Primary Key المعرف الفريد للمستخدم.
username VARCHAR(50) Unique, Not Null اسم المستخدم الفريد لتسجيل الدخول.
email VARCHAR(100) Unique, Not Null البريد الإلكتروني الرسمي.
password_hash VARCHAR(255) Not Null كلمة المرور المشفرة بخوارزمية BCRYPT.
is_active BOOLEAN Default: True حالة تفعيل الحساب في النظام.
failed_login_count INT Default: 0 عدد محاولات تسجيل الدخول الفاشلة المتتالية.
locked_until TIMESTAMP Nullable تاريخ انتهاء قفل الحساب في حالة المحاولات الخاطئة.

3.2 جدول سجلات الطلاب (students)

يحتوي على البيانات الأكاديمية للطلاب بالجامعة.

اسم الحقل (Column) نوع البيانات (Type) القيود (Constraints) شرح الحقل
id UUID Primary Key المعرف الفريد للطالب.
institutional_id VARCHAR(50) Unique, Not Null رقم القيد الجامعي الأكاديمي.
name_ar VARCHAR(150) Not Null اسم الطالب الكامل باللغة العربية.
name_en VARCHAR(150) Not Null اسم الطالب الكامل باللغة الإنجليزية.
enrollment_status VARCHAR(30) Not Null حالة القيد (ACTIVE, SUSPENDED, GRADUATED, UNDER_REVIEW).

3.3 جدول سجلات الموظفين (employees)

يخزن الكادر الأكاديمي والإداري للجامعة.

اسم الحقل نوع البيانات القيود شرح الحقل
id UUID Primary Key المعرف الفريد للموظف.
institutional_id VARCHAR(50) Unique, Not Null الرقم الوظيفي الرسمي للموظف.
name_ar VARCHAR(150) Not Null الاسم باللغة العربية.
name_en VARCHAR(150) Not Null الاسم باللغة الإنجليزية.
employment_type VARCHAR(30) Not Null نوع التوظيف (FACULTY_FULL_TIME, FACULTY_PART_TIME, STAFF).

3.4 جدول المجموعات البحثية (research_groups)

يخزن المجموعات البحثية المسجلة في نظام RGMS.

اسم الحقل نوع البيانات القيود شرح الحقل
id UUID Primary Key المعرف الفريد للمجموعة.
code VARCHAR(30) Unique, Not Null كود التعريفي الفريد للمجموعة البحثية (مثل AI-RG-01).
name_ar VARCHAR(150) Not Null اسم المجموعة بالعربية.
name_en VARCHAR(150) Not Null اسم المجموعة بالإنجليزية.
primary_discipline VARCHAR(100) Not Null التخصص العلمي والبحثي الأساسي للمجموعة.

3.5 جدول انتساب الأعضاء للمجموعات (group_memberships)

ينظم ربط الطلاب والموظفين بالمجموعات البحثية وتحديد مهامهم ونسب مشاركتهم.

اسم الحقل نوع البيانات القيود شرح الحقل
id UUID Primary Key المعرف الفريد للانتساب.
group_id UUID Foreign Key مرتبط بجدول المجموعات البحثية (research_groups.id).
member_uimp_id UUID Foreign Key مرتبط بجدول الطلاب أو الموظفين (المعرف الفريد للمنتسب).
member_type VARCHAR(30) Not Null نوع العضو المنتسب (Staff, Student, External).
role VARCHAR(50) Not Null دور الباحث (PI, Co-I, Research-Assistant, Graduate-Researcher).
workload_percentage INT Check (0-100) نسبة مشاركة الباحث ووقت عمله المخصص للمجموعة.

3.6 جدول سجلات التدقيق الأمني (audit_logs)

يخزن السجلات الأمنية والعمليات الحساسة التي تتم في النظام وهي غير قابلة للتعديل أو الحذف نهائياً.

اسم الحقل نوع البيانات القيود شرح الحقل
id UUID Primary Key المعرف الفريد للعملية الأمنية.
user_id UUID Foreign Key المستخدم الذي قام بالعملية (مرتبط بجدول users.id).
action VARCHAR(100) Not Null نوع العملية (مثل CREATE_STUDENT, UPDATE_ROLE, DELETE_RECORD).
ip_address VARCHAR(45) Not Null عنوان الـ IP للجهاز الذي قام بالعملية.
user_agent TEXT Nullable بيانات المتصفح ونظام التشغيل للمنفذ.
payload JSON Nullable البيانات القديمة والجديدة التي تم تعديلها لتتبع الفروقات بالتفصيل.
created_at TIMESTAMP Not Null وقت حدوث العملية بدقة متناهية.

4. هيكلية وتفصيل الصلاحيات الأمنية ورتب المستخدمين (RBAC Design)

يعتمد النظام بالكامل على نظام الصلاحيات المعتمد على الأدوار (Role-Based Access Control) لإدارة وتوزيع الصلاحيات على المستخدمين بشكل مرن وتلقائي.

4.1 تفصيل الصلاحيات والوصول البرمجي

يتم فحص الصلاحيات أمنياً على مستويين:

  1. على مستوى الواجهات (UI/Blade Level): استخدام وسوم Blade مثل @if(auth()->user()->hasRole('SYSTEM_ADMIN')) لإخفاء أو عرض العناصر والأزرار الحساسة.
  2. على مستوى الخادم والتحقق (Middleware Level): يتم فحص الصلاحية في ملف المسارات لحماية الروابط البرمجية من الوصول المباشر عبر المتصفح.

4.2 حظر وحماية النظام الفرعي للبحوث (RGMS Boundaries)

لمنع تداخل صلاحيات موظفي القبول والتسجيل أو الموارد البشرية بالجامعة مع الأنشطة العلمية والأبحاث، تم برمجة حظر صارم على مستوى المتحكمات (Controllers) والمسارات (Routes) في ملف rgms-subsystem/routes/web.php:

// فحص وحجب المستخدمين غير المصرح لهم بالدخول لنظام البحوث الفرعي
if ($user->hasAnyRole(['REGISTRAR', 'HR_STAFF']) && !$user->hasAnyRole(['SYSTEM_ADMIN', 'UNIVERSITY_ADMIN'])) {
    Auth::logout();
    return back()->withErrors([
        'username' => 'غير مصرح لك بالوصول إلى النظام الفرعي للبحوث (RGMS). / Not authorized to access the RGMS Subsystem.',
    ])->onlyInput('username');
}

5. توثيق واجهات برمجة التطبيقات بالكامل (RESTful API Endpoint Documentation)

يقدم النظام واجهات برمجية كاملة تدعم تبادل البيانات بصيغة JSON القياسية، وهي جاهزة للربط مع تطبيقات الهواتف الذكية أو الأنظمة الأخرى.

5.1 واجهة جلب الطلاب (Get Students List)

تستخدم لجلب بيانات الطلاب النشطين من خلال نظام الجامعة الأساسي.

  • الرابط (Endpoint): GET /api/v1/students
  • الترويس الأمني (Headers): 
    Authorization: Bearer [Sanctum_Token]
Accept: application/json
  • الاستجابة الناجحة (Response 200 OK): 
    {
    "success": true,
    "data": [
        {
            "id": "8a39bd10-23a9-4b6e-8201-44abf01201cc",
            "institutional_id": "ST-2026-009",
            "name_ar": "سراج الخضوري",
            "name_en": "Siraj Alkhaduri",
            "enrollment_status": "ACTIVE"
        }
    ]
}

5.2 واجهة إنشاء مجموعة بحثية جديدة (Create Research Group)

تستخدم لإضافة مجموعة بحثية جديدة في نظام البحوث.

  • الرابط (Endpoint): POST /api/v1/rgms/research-groups
  • الترويس الأمني (Headers): 
    Authorization: Bearer [Sanctum_Token]
Content-Type: application/json
  • محتوى الطلب (Request Payload): 
    {
    "code": "AI-RG-02",
    "name_ar": "مجموعة معالجة اللغات الطبيعية",
    "name_en": "Natural Language Processing Group",
    "primary_discipline": "Artificial Intelligence"
}
  • الاستجابة الناجحة (Response 201 Created): 
    {
    "success": true,
    "message": "Research group created successfully.",
    "group_id": "9a38bd20-3058-45fa-b648-912f205ab090"
}

6. نظام الفحص التلقائي والاختبارات البرمجية (Testing & CI/CD Pipeline)

لضمان جودة الكود البرمجي وخلو النظامين من الأخطاء أثناء التعديل أو التطوير المستمر، تم دمج اختبارات آلية شاملة (Unit & Feature Tests) في المنظومتين.

6.1 تفصيل ملفات الفحص والتحقق الآلي (Tests Suite)

يحتوي كود المشروع على اختبارات تفصيلية تفحص الجوانب التالية:

  • AuditImmutabilityTest: يفحص ويتحقق من استحالة حذف أو تعديل أي سجل في جداول المراقبة الأمنية.
  • AuthLockoutTest: يتحقق من حظر وقفل أي حساب مستخدم بعد 5 محاولات تسجيل دخول خاطئة متتالية كإجراء حمائي ضد هجمات التخمين.
  • RbacBoundaryTest: يتحقق من أن الطلاب لا يمكنهم الوصول لسجلات التدقيق، وأن المدقق لا يمكنه تعديل البيانات، والمسؤول يمكنه التحكم بالكامل.
  • StudentDedupTest: يتحقق من منع إضافة طالب مكرر بنفس رقم القيد لسلامة البيانات.

6.2 التوافقية البرمجية مع محرك الاختبارات (SQLite Compatibility)

نظراً لأن خوادم الاختبارات التلقائية في GitHub Actions تستخدم قاعدة بيانات SQLite الخفيفة بينما تعمل المنظومة الأساسية بمحرك MySQL، قمنا بكتابة قيود وتعبيرات SQL قياسية متوافقة مع كلا المحركين مثل استخدام عبارات CASE WHEN القياسية بدلاً من دوال MySQL الخاصة لضمان ثبات وسلامة عمليات الفحص الآلي واكتمالها بنجاح والحصول على العلامة الخضراء (✅ Passed) دائماً.

7. دليل التنصيب والتهيئة البرمجية للمنظومة (Deployment & Installation Guide)

اتبع التعليمات الهندسية التالية لتهيئة وتشغيل المنظومتين معاً على خادم محلي جديد:

7.1 تهيئة ملفات البيئة والمتغيرات (.env files)

تأكد من إعداد المتغيرات البرمجية في كلا المجلدين uimp-core/.env و rgms-subsystem/.env لربطهما بقاعدة البيانات المشتركة:

# إعدادات قاعدة البيانات المشتركة
DB_CONNECTION=mysql
DB_HOST=127.0.0.1
DB_PORT=3306
DB_DATABASE=uimp_rgms
DB_USERNAME=root
DB_PASSWORD=

# إعدادات مزامنة الجلسات للتزامن الأمني (SSO)
SESSION_DRIVER=database
SESSION_LIFETIME=30

7.2 بناء وتغذية جداول قاعدة البيانات (Migration & Seeding)

لتوليد الجداول وقاموس قاعدة البيانات آلياً مع الحسابات التجريبية، افتح سطر الأوامر داخل المجلد الفرعي لنظام الجامعة uimp-core ونفذ الأمر التالي:

php artisan migrate:fresh --seed

هذا الأمر سيتكفل ببناء كافة الجداول العلائقية وتوليد بيانات الطلاب والموظفين وأعضاء هيئة التدريس الافتراضية مع الصلاحيات المشفرة وكلمة المرور الموحدة password تلقائياً.

7.3 تشغيل المنظومة

اضغط مرتين على ملف start-all.bat في المجلد الرئيسي لتشغيل الخادمين معاً والانتقال مباشرة لفتح الروابط في متصفحك.
